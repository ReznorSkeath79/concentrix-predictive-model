02The Problem We Solved

Why we scrapped STAR and switched to Weighted PTG.

Rejected Approach
STAR Rating
  • Forced bell-curve distribution — top ~10% get STAR 5 regardless of absolute output
  • Bottom ~50% always receive lowest rating — even if entire team improved 20% YoY
  • Model would learn "who is ranked lowest on their team" — not who performs poorly
  • Structurally useless for pre-hire prediction — STAR is a rank, not a measure
VS
Chosen Approach
Weighted PTG (Percentage to Goal)
  • Composite KPI score — weighted aggregate of quality, efficiency, adherence, program KPIs
  • Measured in absolute terms — 95% PTG means the same across every team and program
  • Not forced-distributed — reflects actual performance shape
  • Tercile-based tier labels produce naturally balanced training classes
Cutpoint Derivation

Rather than predicting the raw PTG number, we converted it into tercile-based tier labels. This produces naturally balanced classes (~13,906 agents per tier) — critical for fair model training.

04How the Model Was Built

LightGBM. Bayesian tuning. Cross-validated encoding. Zero leakage.

STEP 01
Data Prep
CDM extract → filter to advisors only → labeled 40,919 → assign tercile tiers → 80/20 train-test split
STEP 02
Encoding
FoldTargetEncoder on high-cardinality fields (Client, Program, Site). Binary flags pass through. NaN handled natively.
STEP 03
Hyperparameter Search
Optuna Bayesian optimization · 50 trials · 5-fold CV on train set · maximize macro AUC
STEP 04
Final Evaluation
Best params → retrain on full 80% → evaluate on held-out 20% → save joblib + schema
Algorithm — LightGBM

Gradient boosting decision tree ensemble. Industry standard for tabular classification.

PropertyHow it helps
Native NaN handlingCandidates with unknown fields are scored without imputation
Mixed feature typesHandles binary, numeric, and categorical in one model
Probability outputsReturns P(At-Risk), P(Developing), P(High Performer)
SpeedFull training in minutes on 40k+ records
No scaling neededTree-based — feature magnitudes don't matter
Leakage Guards

These fields were explicitly excluded — they're either post-hire or would create circular logic:

Excluded — Direct Leakage
Latest Weighted PTG (the target itself) · STAR rating (derived from PTG rankings)
Excluded — Post-Hire Data
Tenure · Performance history · Active-period KPI rows · Manager assignment
FoldTargetEncoder Safety
Out-of-fold statistics used for Client/Program/Site encoding — encoder never sees the current row's label during training
Hyperparameter Optimization — Optuna

Bayesian optimization searches smarter than grid search — it learns which regions of the parameter space are promising and focuses trials there.

ConfigValue
OptimizerOptuna (TPE sampler)
Trials50
CV folds5
MetricMacro AUC (OVR)
Search spacenum_leaves, learning_rate, max_depth, min_child_samples, subsample, colsample_bytree, reg_alpha, reg_lambda

Each trial is fully cross-validated — the chosen parameters generalize reliably and are not overfit to a single train/test split.

Why 5-fold CV?
5 folds means each fold uses 80% of training data to fit and 20% to validate. 50 trials × 5 folds = 250 model fits total. The final best-params model is trained on the full 80% training set once, then evaluated on the held-out 20%.

What the Model Cannot Do
No STAR prediction
STAR is a forced rank. By design, this model does not predict it — and STAR is not a valid performance target.
No manager quality signal
Post-hire factors — team dynamics, manager skill, training quality — are out of scope. The model sees only pre-hire data.
Probabilistic, not deterministic
A "Developing" prediction comes with uncertainty. P(At-Risk) = 35% and P(At-Risk) = 70% both label "At-Risk" but carry very different confidence.
Not a hiring decision engine
This is a risk-ranking tool. Recruiter judgment, structured interviews, and hiring approval remain required.

07Model Improvement Recommendations

10 improvements ordered by impact-to-effort ratio.

High Priority Do these first — low effort, high accuracy gain Medium Lower Priority
01
SHAP Feature Importance on KPI Model
Run SHAP analysis specifically on the KPI model — the current predictor rankings are from deprecated STAR-era models and may not reflect KPI model behavior.
Effort: 1–2 days · SHAP already in codebase
02
Capture the Unlabeled 35.9%
Investigate why 23,322 CDM agents have no PTG value. If they're tenured agents with stable performance, label and include them — more data = more robust model for edge-case demographics.
Effort: Medium · requires data team coordination
03
Probability Calibration
Apply isotonic regression or Platt scaling to calibrate probability outputs. LightGBM probabilities are directionally correct but tend to be overconfident — "P(At-Risk) = 70%" should mean exactly 70%.
Effort: 1 day · scikit-learn CalibratedClassifierCV
04
Program-Specific Sub-Models
Train separate models per high-volume program (Tech Support, Collections, Customer Service). A "High Performer" in Collections looks very different from one in Tech Support.
Effort: 2–3 weeks · needs ≥500 agents per class per program
05
Add SDD1 Data When PTG Available
If the data team backfills Weighted PTG into SDD1, incorporate it into the training set. More diverse training data improves generalization for newer programs and sites.
Effort: Low once data is available
06
6-Month Retraining Cadence
Retrain the model every 6 months using the latest CDM extract. Hiring profiles, program mixes, and performance standards evolve — a 2025 model drifts as the business changes.
Effort: 1–2 days to operationalize as scheduled job
07
Education Level & Assessment Scores
Include educational background and ATS pre-employment assessment scores as additional features. Even a coarse education-level field may add meaningful predictive power.
Effort: Medium · requires ATS data access
08
Time-to-Ramp Prediction
Add a secondary output: predicted time to full productivity alongside tier. A High Performer who takes 6 months to ramp is a different planning case than one performing at Day 45.
Effort: Medium · regression head on same features
09
REST API Wrapper + ATS Integration
Wrap the model as a FastAPI REST endpoint and integrate with the ATS. Recruiters see predicted tier directly on the candidate profile at the moment of review — no CSV export needed.
Effort: 2–3 weeks · FastAPI wrapper + ATS integration
10
Production Feedback Loop
Build a tracking table: every scored candidate → actual PTG outcome at 90–180 days post-hire. Validates production accuracy, surfaces systematic errors by site/program/recruiter, generates labeled data for next retrain.
Effort: Medium · Dataverse table + scheduled join job
